# Delegation Architecture — Trade-offs

**Status:** discussion draft for architectural review
**Date:** 2026-05-10
**Authors:** dev sync — pool/round refactor

---

## 1. Problem statement

Today the codebase has **two distinct delegation patterns** in use, and a third pattern (Tier 2, just shipped) that bypasses both:

1. **Auth-only pattern (existing, for non-on-chain MCP tools).**
   `user.smartAccount → sessionKey` delegation with `TimestampEnforcer` only. Session key (EOA) is generated by a2a-agent at `/session/init`, encrypted, stored. mcp-proxy signs `DelegationTokenClaims` with the session key and forwards a bearer token to MCP servers. MCP `verify-delegation.ts` validates the chain (ERC-1271 against user's smart account). Used by: intents, pledges, proposals, members, beliefs, notifications, private profile, revenue.

2. **Tier 2 D_onchain pattern (just shipped, for pool/round on-chain mutations).**
   A SECOND delegation `user.smartAccount → ORG_MCP_EOA` minted at the same bootstrap, with `Timestamp + AllowedTargets + AllowedMethods` caveats. Stored in a separate cookie. Forwarded as an arg in MCP calls. Org-mcp's wallet (the ORG_MCP_EOA) signs `DelegationManager.redeemDelegation` for pool/round writes.

3. **Direct deployer-signed (legacy, what we just retired).**
   Web action calls `PoolRegistryClient.open(...)` from `getWalletClient()` which is the deployer EOA. Replaced by Tier 2.

The Tier 2 pattern works but it's a side-channel — it duplicates the delegation machinery, splits the key-holder model (a2a-agent holds session key; org-mcp holds redeem key), and is inconsistent with how every other MCP tool gates authority. We want **one delegation pattern across the system.**

The question is what that one pattern should be for the long term.

## 2. Options considered

### Option A — Unified caveat-enforcer composition (single delegation path)

```
USER.smartAccount
    │
    │ ONE delegation, signed at /session/package
    │ delegate = sessionKey (EOA in a2a-agent)
    │ caveats = [Timestamp, AllowedTargets, AllowedMethods,
    │            ValueEnforcer, McpToolScope, RateLimit]
    ▼
A2A AGENT (holds encrypted sessionPrivateKey)
    ├─ MCP tool auth: sign DelegationTokenClaims → MCP bearer
    └─ On-chain redeem: new POST /session/{id}/redeem-tx endpoint
       submits DelegationManager.redeemDelegation([delegation], target, value, data)
       with session key as msg.sender

MCP TOOL (org-mcp, person-mcp, …)
    ├─ Authenticate via existing verify-delegation.ts
    ├─ SQL-side ops: as today
    └─ On-chain ops: POST to a2a-agent's redeem endpoint
```

**Key properties:**
- One delegation. One key holder (a2a-agent). One signing path.
- All on-chain policy enforced by composed caveat enforcers (modules in this codebase).
- 15 enforcers already exist; we use 5-6 of them on the session delegation.

### Option B — Modular session AgentAccount (ERC-7579-style)

```
USER.smartAccount
    │
    │ delegates to sessionAccount (deployed AgentAccount per session)
    │
    ▼
SESSION AGENT ACCOUNT (deployed smart contract per session)
    ├─ Validator module: ECDSA over session key
    ├─ Executor module: routes calls to allowed targets
    ├─ Hook modules: spend tracker, rate limiter, target allowlist
    │  (modules can hold state across calls — e.g., remaining USDC budget)
    │
    └─ ERC-7579 install/uninstall API for runtime composition

A2A AGENT
    holds sessionPrivateKey (validator-recognized)
    submits userOps through EntryPoint → sessionAccount → target

MCP TOOL
    receives sub-delegation from sessionAccount per call OR
    submits to a2a-agent which routes through sessionAccount
```

**Key properties:**
- Session is its own deployed smart account.
- Policy is enforced **inside the account** via modules — not at delegation-redeem time.
- Modules can persist state (a critical capability for spend tracking, usage quotas, etc.).
- Composable runtime — install a module to add a capability without redeploying the account.

### Option C — Chained delegation with per-call sub-delegations

```
USER.smartAccount
    │
    │ D_root: delegate = sessionAccount (broad-but-bounded, task-bound)
    │
    ▼
SESSION (AgentAccount or EOA)
    │
    │ For each MCP tool call, mint D_sub:
    │   delegate = mcpToolExecutor (per-tool identity)
    │   authority = hash(D_root)
    │   caveats = exact target, exact selector, single-use nonce, value cap
    │
    ▼
MCP TOOL EXECUTOR (own EOA per tool family)
    redeemDelegation([D_sub, D_root], target, value, data)
    msg.sender = toolExecutor (= leaf delegate)
```

**Key properties:**
- Two-element chain on every redeem. DelegationManager validates leaf→root.
- Per-call narrower authority. Defeats "session key leak → drain everything in allowlist" because each call needs a fresh sub-delegation that the session must sign.
- MCP tool has its own cryptographic identity — auditable per-tool.

### Option D — Hybrid: A + targeted C

```
Default path = Option A (caveat composition)
Promoted path = Option C (per-call sub-delegation) for high-value ops

Triggered by a flag on the tool definition:
  - pool:create → caveat-composition
  - round:set_awards_root → sub-delegation chain (more sensitive)
```

## 3. Evaluation dimensions

| Dimension | Why it matters long-term |
|---|---|
| **Stateful policy** | Some limits (spend caps, usage quotas) require state across calls. Caveats are stateless; modules can persist. |
| **Audit correlation** | Long-term: every action traceable from user signature → A2A task → MCP call → on-chain tx. Chain depth helps; module hooks help even more. |
| **Per-tool blast radius** | If one MCP tool is compromised, how much can it do? Lower blast = better. |
| **Key holder count** | More services holding signing keys = more attack surface. |
| **Runtime composability** | Can we add a new policy (e.g., a geo-fence) without redeploying every existing session? |
| **Standards alignment** | Future interop with wallets / agents / tools. ERC-4337, ERC-7579, ERC-7710, ERC-7715. |
| **Gas / cost per session** | Per-session AgentAccount deployment is non-trivial (CREATE2 + module install). Caveats are read-only at validate time. |
| **Implementation complexity** | Weeks-of-work estimate plus subtle-bug risk. |
| **Revocation granularity** | Revoke whole session vs. revoke specific tool's sub-delegation. |
| **Cross-MCP coordination** | Does the design naturally extend to person-mcp / family-mcp / new MCPs? |

## 5. Subtle considerations

### What does "stateful policy" cost us in Option A?

Caveats are pure functions: `(delegation, target, value, data) → bool`. They can't track "you've already spent 5 USDC, now you have 20 left." The closest we get with caveats is `RateLimitEnforcer` (calls/window) and `AllocationLimitEnforcer` (which uses pool state, not delegation state).

If we want per-session spend tracking, we have two paths:
- **(A1)** External mutable state via an enforcer that reads/writes a side contract (e.g., `SessionSpendTracker`) — keeps caveat composition but adds a state contract per session.
- **(B)** Native module storage on the session AgentAccount — modules read/write account storage directly.

(A1) is implementable in Option A by writing a new enforcer. (B) is more natural in Option B but requires the modular account.

### What does "per-call sub-delegation" actually buy us?

The big win is **single-use nonces**. With Option A's session-wide allowlist, if the session key is stolen the attacker can issue N redeemDelegations within the rate limit. With Option C's `[D_sub, D_root]` chain, each sub-delegation has a single-use nonce; attacker needs the session to sign a fresh sub-delegation for each call. If we revoke the session, all future sub-delegations fail.

The cost: every MCP-tool-to-on-chain hop requires a session-side signature. That means a2a-agent (which holds the session key) is in the loop on every call. In Option A it's also in the loop (for the redeem submit), so the cost is similar; mainly it's one extra signing event per call.

### What does Option B (modular account) actually give us beyond Option A?

The unique capability is **state in modules**. A `SpendCapHookModule` can:
- On install: set `budget = 100 USDC`
- On every call: `require(spent + value <= budget); spent += value`
- Persists across calls without external lookups

You can't do this with a stateless caveat. You CAN do it with `SessionSpendTracker` external contract referenced by a caveat (variant A1 above), but that's a more roundabout shape than "the account itself knows its budget."

The OTHER thing Option B gives: **runtime install/uninstall.** Add a `GeoFenceModule` to an active session without re-signing. Useful if policy evolves during a long-running session.

### When does Option C's audit trail matter?

Today every redeem emits a `DelegationRedeemed(hash, delegator, delegate)` event. In Option A you get one event per session per chain. In Option C you get one per *call* — the leaf delegation hash is unique per call. For forensics ("what tool, on what call, on what task, signed by what session, on what root grant?"), Option C produces a richer audit trail at zero extra cost beyond the EIP-712 sign.

### Cross-MCP authority overlap

If person-mcp wants to write on chain (e.g., a person-side `intent:publish` that mints an `sa:IntentAssertion`), we need either:
- A's path: same caveat composition lets person-mcp also use the session delegation (just add the relevant target/selectors to AllowedTargets/AllowedMethods).
- C's path: person-mcp gets its own tool-executor identities.

Either way works. Option A scales by enlarging the caveat list; Option C scales by minting more executor identities. For long-term cross-MCP architecture, Option A's "one big allowlist per session" centralizes policy; Option C's "many small executors" distributes it. The distributed approach is more zero-trust-friendly but more complex operationally.

## 6. Long-term considerations

### Forecast: what will this codebase need in 12 months?

Based on the trajectory (intent marketplace, pool/round/proposal/match flows, expanding to engagement entitlements, treasury custody, agent communication via A2A):

- **Spend caps that persist:** Yes (treasury Phase 3, pool allocations, disbursements). Argues for Option B's stateful modules OR Option A's external state contracts.
- **Per-tool identity & audit:** Yes (compliance, third-party agent execution). Argues for Option C.
- **Runtime policy changes:** Maybe — useful for "user adds geo-restriction mid-session." Mostly Option B.
- **Cross-MCP execution:** Yes (person-side intents, family/skill/geo MCPs all eventually writing on chain). Argues for the unified-session approach (Option A's strength).
- **Long-lived sessions for AI agents:** Yes — autonomous agents will hold sessions across days/weeks. Argues for stateful policy (B/C) over stateless caveats (A).

### Forecast: what does "best long-term" actually mean?

If "best" = **secure, auditable, composable, future-proof**: Option D (Hybrid) wins, with the **modular session AgentAccount (B) as the eventual target** for stateful policy, and **per-call sub-delegations (C)** layered on for high-value operations.

If "best" = **shipping in 2 weeks while not painting ourselves into a corner**: Option A is the right starting point — it eliminates the Tier 2 side-channel, uses existing infrastructure, and is a strict superset of what we have today. Migration to B/C later is additive (the session-delegation pattern continues to work; modules and sub-delegations are layered on).

## 7. Recommendation

**Adopt a 3-phase target:**

### Phase 1 — Unify on Option A (caveat-enforcer composition) — **execute first**
- Drop Tier 2's D_onchain side-channel.
- Pool/round move to the existing session-key delegation path with enriched caveats: `Timestamp + AllowedTargets + AllowedMethods + ValueEnforcer + McpToolScope + RateLimit`.
- New a2a-agent endpoint `POST /session/{id}/redeem-tx` for on-chain submission.
- Org-mcp loses its wallet; becomes pure coordinator.
- Net change: one delegation pattern, 5-6 enforcers composed, single key holder.
- Estimated effort: 1 week.

### Phase 2 — Per-call sub-delegations for high-value ops (Option C, selective)
- Add `TaskBindingEnforcer` contract.
- Promote sensitive operations to per-call sub-delegation: `round:set_awards_root` (controls who gets grants), `disbursement:claim` (controls treasury outflow), `pool:close`, future treasury / custody ops.
- Each "promoted" tool family has its own executor EOA. Routine ops (intents, pledges, low-value pool admin) stay on Phase 1 path.
- Audit trail strength: per-call sub-delegation hashes. Replay defense: single-use nonces.
- Estimated effort: 2-3 weeks.

### Phase 3 — Modular session AgentAccount (Option B) — for stateful policy
- Trigger: when we ship treasury custody, multi-call spend caps, or runtime-policy updates.
- Implement ERC-7579 install/uninstall on AgentAccount (currently introspection-only per the contract comment).
- Build 3-5 first-party modules: `SpendCapHook`, `GeoFenceHook`, `MultiSigValidator`, `SessionExpiryModule`.
- Per-session deploy flow (CREATE2 + module install) in a2a-agent's `/session/init`.
- Existing caveat-composition path coexists for sessions that don't need stateful policy.
- Estimated effort: 4-6 weeks.

**Rationale:** Phase 1 ships value fast and corrects the current Tier 2 inconsistency. Phase 2 raises the security floor on critical operations without forcing every tool through the heavier path. Phase 3 unlocks long-running autonomous sessions with persistent policy — which is the eventual must-have for AI agents holding sessions across days.

Each phase is **additive** — none of them throw away the previous phase's work. The session delegation pattern from Phase 1 keeps working in Phase 2; Phase 2's per-call hashes coexist with Phase 1's session-level audit; Phase 3's modular accounts can either replace EOA session keys session-by-session or coexist forever.

## 8. Open questions for review

1. **Spend tracking urgency.** Is treasury Phase 3 close enough (months) that we should consider B sooner, or far enough out that A is fine for now?
2. **Per-tool executor identity.** Even within Phase 1, do you want separate executor EOAs per tool family (pool, round, intent, pledge, disbursement) for audit clarity — or one shared session executor?
3. **TaskBindingEnforcer scope.** Phase 2 introduces this. Should "task" be A2A task ID, or something coarser (a "session intent" the user approved in the UI)? The former gives stronger correlation; the latter is simpler to UX.
4. **Cross-MCP allowlist size.** Phase 1's session caveat lists all allowed methods. With 5+ MCPs eventually writing on chain, the list grows. Acceptable, or do we want per-MCP sub-sessions?
5. **Revocation UX.** Phase 1 revokes whole sessions. Phase 2 revokes specific sub-delegation hashes. Do you want both surfaces in the UI, or just "log out this session"?
6. **Standards stance.** ERC-7710 (delegation) and ERC-7715 (wallet permissions) are still drafts. Comfortable depending on them, or design as if they could change?
