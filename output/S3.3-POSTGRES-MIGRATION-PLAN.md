**Synthesis date**: 2026-05-17
**Status**: SPEC ONLY. No code in this PR series until §10 PR-1 lands.
**Inputs**: senior security review (multi-instance state-consistency finding), Sprint 2 S2.1 sub-agent note ("cross-process TOCTOU remains possible if multiple person-mcp instances ever share the same SQLite file"), Sprint 3 S3.1 audit-checkpoint surface (`apps/a2a-agent/src/db/schema.ts:153`), current SQLite call-sites in `apps/a2a-agent/src/db/`, `apps/person-mcp/src/db/`, `apps/person-mcp/src/session-store/`, `apps/web/src/db/`.

## 1. Why this matters

Every hardening promise the system has made since Sprint 1 — replay-nonce uniqueness, action-counter caps, idle deadlines, revocation-epoch monotonicity, the append-only audit chain, the Sprint 3 S3.1 signed checkpoint — assumes that the relevant table is the **single, strongly consistent source of truth** for the property being enforced. That assumption holds today because each backend service (a2a-agent, person-mcp, org-mcp, hub-mcp, web) runs as a single Node process pointing at a single local `*.db` file. `better-sqlite3` is synchronous; its writer lock is process-local; its WAL serializes writes to a single file pointer. One process, one writer, one consistent view.

The moment any of these services is horizontally scaled — a Vercel deployment that fans out to two function instances, a Kubernetes Deployment with `replicas: 2`, a blue/green cutover overlap, even a Vercel "warm + cold" pair handling concurrent requests — the assumption shatters. Two instances cannot share a SQLite file safely: the WAL writer-lock window is process-local, so two concurrent `consumeAction` transactions on instances A and B can both pass the SELECT, both UPDATE, and both commit, leaving the counter at +1 instead of +2 — the exact TOCTOU race Sprint 2 S2.1 closed for the single-process case but explicitly tagged out-of-scope for the multi-process case. A session minted on instance A is not visible to instance B until the file system replicates the row (it doesn't); a nonce burned on instance A is not seen by instance B (it isn't); a revocation-epoch bump on instance A is invisible to instance B; an audit row written on instance A interleaves with one on instance B but the hash-chain `prev_entry_hash` is computed against the local view, so two divergent forks of the chain co-exist with no way to merge them. The S3.1 signed checkpoint, signed against a local fork, attests to a history that never globally existed.

S3.3 migrates the **system-state tables** (sessions, audit, action counter, replay nonce, revocation epoch, idle-deadline state, audit checkpoint) to Postgres so that horizontal scaling becomes a configuration change, not a security regression. User-data tables (profile, chat history, oikos contacts, prayers, etc.) migrate alongside for operational reasons (one DB per service, not a mix) but their concurrency model is unchanged — Postgres handles them with default `READ COMMITTED` semantics exactly as SQLite did. The load-bearing change is the move from per-process write-lock serialization to **per-row Postgres row locks** under `SERIALIZABLE` isolation for the five hot paths: `consumeAction`, `consumeActionNonce`, `bumpRevocationEpoch`, `appendAuditEntry` (chain head), `interServiceNonce.insert`. Everywhere else the migration is mechanical schema-and-driver swap.

The non-goal worth stating up front: this is **not** a "make SQLite scale to multiple instances" project. SQLite's design forbids that and any attempt would be a security smell. This is **swap the substrate** so the existing invariants hold under N>1 instances.

## 3. Migration strategy

### 3.1 ORM: stay on Drizzle

The codebase already uses Drizzle for `apps/web` and `apps/a2a-agent`; the others use raw `better-sqlite3.prepare` calls but read/write the same logical schema. Drizzle's killer feature for this migration is **dialect-portable schema definitions**: a single `sqliteTable(...)` definition becomes a `pgTable(...)` definition with the same column names. The query builders are dialect-aware (`db.select().from(...)`, `db.insert(...).onConflictDoNothing()`) so call-sites that already use the builder need almost no change. Raw `.prepare(...)` callsites in person-mcp/org-mcp/hub-mcp will be migrated to the builder during PR-3 — this is half the work of S3.3.

We stay on Drizzle. No ORM swap. (§15 reinforces this.)

### 3.2 Driver

- SQLite: `better-sqlite3` + `drizzle-orm/better-sqlite3` (today).
- Postgres: choose **`@neondatabase/serverless`** for Vercel deploys (HTTP-mode, no long-lived TCP socket — survives the Vercel function lifecycle), with `drizzle-orm/neon-http` (or `neon-serverless` for transactions). For local dev and self-hosted runtimes, `postgres` (the `postgres-js` driver by Porsager) with `drizzle-orm/postgres-js`. Both are first-class in Drizzle. The selector is environment-driven (§10 PR-1).
- `pg` (node-postgres) is the boring third option — fine for self-hosting but slower cold-start than `postgres-js`.

### 3.3 Three deployment models

| Model | Pros | Cons | Recommendation |
|---|---|---|---|
| **Managed Postgres** (Neon, Supabase, RDS) | Zero ops; PITR baked in; branching for preview deploys (Neon) | Vendor lock-in at the connection-string level | **Recommended — Neon** |
| **Vercel Postgres** (now Marketplace-fronting Neon) | One-click from the dashboard; auto-injected `DATABASE_URL`s; OIDC handshake | Marketplace integration matures Q-by-Q; pricing slightly above Neon-direct | Acceptable alternate; identical schema |
| **Self-hosted Postgres** | Maximum control; on-prem option | Patching, backups, HA failover, monitoring, PITR all your problem; not justified for current scale | Out for v1 |

**Pick: Neon, accessed through the Vercel Marketplace integration** so production `DATABASE_URL`s are auto-injected at build time, preview deploys get branched databases for free (Neon's branch-per-PR is exactly the workflow the spec series has been pushing), and the failure mode if we want to leave is "export + import to RDS." Free tier (0.5 GB) covers dev; pro tier (~$20/mo per database) covers staging; production starts at ~$50/mo and scales by storage + compute.

### 3.4 Translation cheat sheet (SQLite → Postgres)

| SQLite | Postgres | Notes |
|---|---|---|
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGSERIAL PRIMARY KEY` or `GENERATED ALWAYS AS IDENTITY` | Drizzle emits `serial()` / `bigserial()` |
| `TEXT` | `TEXT` | Identical |
| `INTEGER` | `BIGINT` | The audit-log `ts_ms` is a unix ms timestamp; needs 64-bit |
| `INTEGER` (boolean) | `BOOLEAN` | Migrate `report_required` 0/1 → false/true at translation |
| JSON in TEXT | `JSONB` | `recent_timestamps`, `grant_json`, `delegation_json`, `terms` — all become JSONB for indexable + atomic-update paths |
| `INSERT OR IGNORE` | `INSERT ... ON CONFLICT DO NOTHING` | Idempotent inserts |
| `INSERT OR REPLACE` | `INSERT ... ON CONFLICT (k) DO UPDATE SET ...` | Upsert |
| `BEGIN IMMEDIATE` | `BEGIN ISOLATION LEVEL SERIALIZABLE` (selectively) | See §5 |
| `PRAGMA journal_mode = WAL` | n/a | Postgres uses WAL by default |
| `PRAGMA table_info(t)` | `information_schema.columns` | Used in idempotent ALTER blocks — see §10 |
| `json_extract(c, '$.k')` | `c->>'k'` (text) or `c->'k'` (jsonb) | None currently used; `recent_timestamps` is parsed in app code so this is a no-op for now |
| `datetime('now')` | `now()` | Few callsites; mostly we pass ISO strings from app |

### 3.5 Idempotent schema bootstrap

The current SQLite story bootstraps schema in `db/index.ts` with `CREATE TABLE IF NOT EXISTS` plus best-effort `ALTER TABLE ... ADD COLUMN` blocks wrapped in try/catch (see `apps/a2a-agent/src/db/index.ts:116-157`, the K0+K1 KMS migration block, the Phase 4 `users → local_user_accounts` rename in `apps/web/src/db/index.ts:34`, the data-store-consolidation `DROPPED_TABLES` list).

That hand-rolled migration story does **not** survive the move to Postgres. We replace it with `drizzle-kit migrate`:
- Schema lives in `src/db/schema.ts` (already does).
- `pnpm drizzle-kit generate` produces a numbered SQL file in `apps/<svc>/drizzle/` per change.
- `pnpm drizzle-kit migrate` (or app-startup hook) applies pending migrations in order, tracked in Postgres's `drizzle.__drizzle_migrations` table.
- The K0+K1 ALTER blocks become first-class migration files. The `DROPPED_TABLES` cleanup runs once during the cutover migration and is then deleted from the boot path.

`fresh-start.sh` already wipes per-service SQLite files; in the Postgres world it `psql -c "DROP DATABASE … CASCADE"` then re-applies migrations.

## 4. Per-table migration details

For each load-bearing table I list (a) schema deltas, (b) the post-migration concurrency model, (c) how tests change.

### 4.1 `sessions` (person-mcp — the canonical SessionRecord)

**Schema deltas**:
- `created_at_ms`, `expires_at_ms`, `idle_expires_at_ms`, `revoked_at_ms` stay as `BIGINT` (ms-since-epoch). We considered moving to `TIMESTAMPTZ` but the wire format inside the SessionGrant is numeric ms — translating in two places is fragile.
- `grant_json` becomes `JSONB`. Reads still go `JSON.parse(row.grant_json)` in app code for now; a future PR can use `->>` to project specific fields. Keeping the app code dialect-blind is the v1 goal.
- `session_id_hash` keeps its `UNIQUE` constraint (matches the cookie-value lookup pattern).
- New index: `CREATE INDEX idx_sessions_active ON sessions (smart_account_address) WHERE revoked_at_ms IS NULL AND expires_at_ms > now_ms`. Partial index, narrows the `listActiveSessionsForAccount` hot path.

**Concurrency model**: READ COMMITTED for all reads/writes. The `bumpIdleDeadline` write is a single `UPDATE sessions SET idle_expires_at_ms = ? WHERE session_id = ?` — no race, the last writer wins; idle deadline is monotonic in the application layer (callers always pass `max(current, new)` semantics via the verify path).

**Tests**: `apps/person-mcp/test/session-store.test.ts` (and similar) move from in-process SQLite to a Postgres test backend. See §13.

### 4.2 `revocation_epochs`

**Schema deltas**: `smart_account_address` `TEXT PRIMARY KEY`, `epoch` `BIGINT NOT NULL`.

**Concurrency model**: The current SQLite path tries `UPDATE … SET epoch = epoch + 1`, falls back to `INSERT` if no row. In Postgres this becomes a single upsert with a returning clause:

```sql
INSERT INTO revocation_epochs (smart_account_address, epoch)
VALUES ($1, 1)
ON CONFLICT (smart_account_address)
  DO UPDATE SET epoch = revocation_epochs.epoch + 1
RETURNING epoch;
```

Atomic, single statement, cross-instance correct. No SERIALIZABLE needed — the row lock that `ON CONFLICT … DO UPDATE` takes is exactly what we want.

**Tests**: race test — N parallel `bumpRevocationEpoch` for the same account, assert the final `epoch` equals N exactly.

### 4.3 `action_nonces_v2` and `inter_service_nonce` (replay nonce caches)

**Schema deltas**: `nonce_key TEXT PRIMARY KEY` (resp. `nonce TEXT NOT NULL UNIQUE`).

**Concurrency model**: Postgres UNIQUE constraint + `INSERT ... ON CONFLICT DO NOTHING RETURNING 1`. The `RETURNING` row is non-empty iff the insert succeeded; empty `RETURNING` means another instance already burned this nonce.

```ts
const inserted = await tx.execute(sql`
  INSERT INTO action_nonces_v2 (nonce_key, consumed_at_ms, expires_at_ms)
  VALUES (${key}, ${now}, ${now + NONCE_TTL_MS})
  ON CONFLICT (nonce_key) DO NOTHING
  RETURNING 1
`)
if (inserted.rowCount === 0) throw new Error('action nonce already consumed (replay rejected)')
```

This is the canonical cross-instance replay-protection pattern; Postgres serializes the UNIQUE index contention internally.

**Tests**: spawn two pg connections in the same test, race both at the same `(account, actionNonce)`, assert exactly one succeeds. Today's `apps/person-mcp/test/action-counter.test.ts` "concurrent verifies" test (lines 224-253) becomes the template.

**GC**: opportunistic GC (`DELETE … WHERE expires_at_ms < now`) on every insert is wasteful at scale. Replace with a per-service `pg_cron`-style job or a Vercel cron at `/internal/gc-nonces` running every 5 minutes.

### 4.4 `session_action_count` (S2.1 — the load-bearing TOCTOU case)

This is the one the S2.1 sub-agent explicitly flagged. See §5 for the full SQL pattern. Schema deltas:

- `total_actions BIGINT`.
- `recent_timestamps` becomes `JSONB` (was TEXT-encoded JSON in SQLite). The trim-and-append logic moves out of app code into a single SQL statement, or stays in app code reading/writing `JSONB` — we keep it in app code for v1 (less SQL surface to reason about) and revisit if profiling demands a SQL-side trim.

### 4.5 `execution_audit` and `audit_log` (the hash chain)

**Schema deltas**:
- `id BIGSERIAL PRIMARY KEY` for `execution_audit`, `seq BIGSERIAL PRIMARY KEY` for `audit_log`. Postgres sequences give us monotonically increasing IDs that are unique across instances (the sequence is database-global). This is what the chain needs.
- `received_at`, `finalized_at`, `ts_ms` stay as `TEXT` / `BIGINT` (already do).
- New index for the S3.1 checkpoint: `CREATE INDEX idx_execution_audit_id_desc ON execution_audit (id DESC)`. The checkpoint reader does `SELECT … ORDER BY id DESC LIMIT 1`.

**Concurrency model**: READ COMMITTED for the `INSERT`. The hash-chain `prev_entry_hash` is computed by reading "the latest row for this account" and hashing into the new row. Under multi-instance writes, two `appendAuditEntry` calls for the same account can race; both read the same `prevEntryHash`, both compute hashes, both insert. The chain forks: two rows both claim `prev = X` and there's no recovery.

**Fix**: take a SERIALIZABLE transaction for the read-and-append. Postgres detects the read-write conflict and aborts one of the two with `40001` `serialization_failure`. The caller retries. See §7 for the exact SQL.

**Append-only invariant** (already enforced by `scripts/check-no-bypass.sh` against `db.update(executionAudit)` and `db.delete(executionAudit)`) carries over — the script's grep target changes from `'drizzle-orm/sqlite-core'` callsites to `'drizzle-orm/pg-core'`, but the lint stays.

### 4.6 `audit_checkpoint` (S3.1)

**Schema deltas**: `signature TEXT NOT NULL` (hex), `signer_address TEXT NOT NULL`, the rest unchanged. `latest_entry_id BIGINT` (matches the bigserial it references).

**Concurrency model**: Single writer per service (the cron). Reads from the sink uploader. No race.

**New cross-instance invariant**: with N instances, only one should emit checkpoints. Use `pg_try_advisory_lock` keyed on a constant ID for the checkpoint job; the loser becomes a no-op. Trivial 3-line change in the cron entry point.

### 4.7 `challenges`, `handles`

**Schema deltas**: trivial. `TEXT PRIMARY KEY` stays, `UNIQUE` indexes stay.

**Concurrency model**: READ COMMITTED.

### 4.8 web tables

`local_user_accounts`, `recovery_delegations`, `recovery_intents`, `invites`, `training_modules`, `detached_members` — all schema deltas are mechanical (`INTEGER` → `BIGINT` where applicable; the rest stays). `recovery_intents` gets a SERIALIZABLE check-and-insert (same pattern as nonces) to ensure the timelocked-recovery `intent_hash` UNIQUE constraint is enforced cross-instance.

### 4.9 person-mcp / org-mcp user-data tables

The 25-ish user-data tables listed in §2.2/2.3 all migrate with default isolation. No special treatment. They become Drizzle `pgTable` definitions and the existing raw-`.prepare` SQL is rewritten to use the Drizzle query builder during PR-3.

## 5. The action-counter cross-process TOCTOU fix

The S2.1 sub-agent's exact concern, re-stated: today `consumeAction` (`apps/person-mcp/src/session-store/index.ts:271-337`) opens a `better-sqlite3` synchronous transaction. Inside it, it `SELECT`s the counter row, evaluates the cap, and `UPDATE`s. Because `better-sqlite3` is synchronous and the SQLite file holds a single writer lock for the WHOLE transaction duration, two concurrent verifies in the same process serialize: the second sees the first's increment before evaluating. The race is closed for one process.

Two processes sharing the same SQLite file is **not** safe. SQLite's WAL gives readers MVCC visibility but writers still take an exclusive lock for the BEGIN-IMMEDIATE → COMMIT window. Two processes can both call `BEGIN IMMEDIATE`; one wins, the other gets `SQLITE_BUSY` and retries. In practice this can degrade to lost updates if the busy-timeout fires inside the transaction. Even when it works, it's not the contract we want — we want "two concurrent verifies always see each other's writes," and SQLite multi-process is too brittle to lean on.

Postgres pattern, post-migration:

```sql
BEGIN ISOLATION LEVEL SERIALIZABLE;

SELECT total_actions, recent_timestamps
  FROM session_action_count
  WHERE session_id = $1
  FOR UPDATE;       -- row lock for the rest of the txn

-- 1. If no row, INSERT new row with total_actions=1, recent=[now].
-- 2. If row exists, parse recent_timestamps, drop expired, evaluate caps,
--    decide allowed/denied. If allowed, UPDATE total_actions, recent_timestamps, updated_at_ms.

COMMIT;
```

`SELECT ... FOR UPDATE` is the explicit row-level lock — Postgres holds it until the transaction commits or rolls back. Any concurrent transaction touching the same row blocks at its own `SELECT ... FOR UPDATE`. Two instances racing on the same `session_id` serialize: the second sees the first's UPDATE after the first COMMITs, exactly as the single-process SQLite case does today.

Why `SERIALIZABLE` and not just `READ COMMITTED + FOR UPDATE`? Two reasons:
1. **Defense-in-depth**: SERIALIZABLE adds Postgres's predicate-locking. If a later schema change adds a path that reads the counter without `FOR UPDATE` (e.g. the `getActionCounter` inspector), SERIALIZABLE catches the conflict; READ COMMITTED+`FOR UPDATE` only locks the rows you explicitly took.
2. **Symmetric with the audit chain**: `appendAuditEntry` also wants SERIALIZABLE (see §7). One isolation level for hardening-critical paths is easier to reason about than two.

Cost: SERIALIZABLE serialization conflicts surface as `40001` errors. The caller (`verifyDelegatedWalletAction`) wraps `consumeAction` in a small retry loop — 3 attempts, exponential backoff up to 50ms, then surface as `action-cap-exceeded` (semantically: contention means the cap is being actively contended; deny is safer than allow). The retry helper goes in `packages/sdk/src/db/serializable-retry.ts`.

Test (the race test from `action-counter.test.ts:224`): spawn 5 parallel `consumeAction` against a cap of 2 *across two pg connections*. Assert exactly 2 succeed. The existing test (lines 224-253) becomes the template; the Postgres version uses `pg.Pool` with `min: 2, max: 2` to force two distinct backend connections.

## 6. The replay-nonce concurrency model

Both `inter_service_nonce` (a2a-agent + each MCP) and `action_nonces_v2` (person-mcp delegation actions) are uniqueness checks. The canonical Postgres pattern, already shown in §4.3:

```ts
const row = await tx.execute(sql`
  INSERT INTO inter_service_nonce (nonce, service, used_at)
  VALUES (${nonce}, ${service}, ${new Date().toISOString()})
  ON CONFLICT (nonce) DO NOTHING
  RETURNING 1
`)
if (row.rowCount === 0) throw new ReplayDetected()
```

Postgres serializes UNIQUE-constraint contention at the index level; multiple inserts of the same nonce from N instances will result in exactly one success and N-1 `DO NOTHING` outcomes. The `RETURNING 1` is the witness — present iff this caller's insert is the one that won. No isolation override needed (the unique index enforces atomicity).

Note: SQLite today throws `SQLITE_CONSTRAINT_UNIQUE` and the caller catches by message-matching the string `UNIQUE constraint`. Postgres returns no error in `ON CONFLICT DO NOTHING` mode — we check `rowCount`. The translation lives in `consumeActionNonce` / `requireInboundServiceAuth` and is part of PR-3.

GC: same `/internal/gc-nonces` cron pattern as §4.3.

Per-service vs shared `inter_service_nonce` table: today each backend service has its own `inter_service_nonce` table in its own SQLite. Post-migration this stays the same — per-service Postgres database (recommended, §11), each holding its own `inter_service_nonce`. We do NOT centralize nonce caches across services. A replay attempt is service-scoped (a nonce is generated by the caller, sent to one target service, and burned there); the target service is authoritative for its own nonce table. This matches today's logical model.

## 7. Audit chain cross-instance

S3.1's signed checkpoint becomes load-bearing the moment N>1. Without a globally-ordered chain, the signature on `latest_entry_hash` is signing one fork — the SIEM sees one history, the local DB another. Reconciliation requires merging the forks, which the chain construction (`prev_entry_hash` linking) does not support.

**Fix**: the `appendAuditEntry` write becomes a SERIALIZABLE transaction:

```sql
BEGIN ISOLATION LEVEL SERIALIZABLE;

SELECT entry_hash FROM audit_log
  WHERE smart_account_address = $1
  ORDER BY seq DESC LIMIT 1
  FOR UPDATE;     -- locks the latest row; new INSERTs serialize behind it

-- App code computes entry_hash = sha256(prev || canonical(input))

INSERT INTO audit_log (
  ts_ms, smart_account_address, session_id, grant_hash,
  action_id, action_type, action_hash,
  decision, reason, audience, verifier,
  prev_entry_hash, entry_hash
) VALUES (...) RETURNING seq;

COMMIT;
```

`FOR UPDATE` on the most-recent row prevents two concurrent appends from both reading the same `prev_entry_hash`. The second waits, then re-reads (now seeing the first's INSERT), and chains correctly. The `seq` BIGSERIAL gives us the global total order — `ORDER BY seq` returns events in commit order, and Postgres assigns sequence values BEFORE the lock is released, so the order is gap-free in commit order (sequences themselves can have gaps from rolled-back transactions, but the relative order within COMMITs is preserved).

If two appends conflict on SERIALIZABLE, the loser gets `40001` and the caller (in `lib/audit.ts auditAppend()`) retries. The retry helper is the same one §5 uses.

`execution_audit` in a2a-agent is the same shape — `appendAuditEntry` for the per-action redeem audit becomes a serializable read-modify-write iff the chain is per-(grant, account); the current schema's `id INTEGER PRIMARY KEY AUTOINCREMENT` doesn't have a hash chain (the chain lives in person-mcp's `audit_log`), so its INSERTs are pure appends and need no SERIALIZABLE wrapping. The Bigserial sequence assigns ordering.

S3.1 checkpoint signer reads `SELECT id, entry_hash FROM execution_audit ORDER BY id DESC LIMIT 1` and signs `keccak256(latestEntryHash || timestamp || chainId)`. With Postgres ordering, this is the single global head. The `pg_try_advisory_lock` mentioned in §4.6 ensures only one instance signs per interval.

## 8. Connection pooling

Vercel functions are short-lived; opening a TCP-backed Postgres connection per cold start is expensive (~50-100ms). Two options:

**Option A — Neon HTTP driver** (`@neondatabase/serverless` with `drizzle-orm/neon-http`): every query is a single HTTP request over Neon's connection-pooling proxy. No connection lifecycle to manage; zero per-request cost beyond the HTTP overhead. The constraint: **no transactions across HTTP requests**. Multi-statement transactions need the `neon-serverless` (WebSocket) driver instead.

**Option B — `postgres-js` with PgBouncer-style pooling** (Neon offers it transparently; `@neondatabase/serverless` falls back to PG protocol over WebSockets when transactions are needed): `drizzle-orm/postgres-js`. Per-instance pool, `max: 10` is the canonical Vercel function size. Cold start opens the pool lazily; warm starts reuse.

**Recommended stack**:
- a2a-agent + person-mcp + org-mcp + hub-mcp (Hono servers, persistent processes): `postgres-js` with `max: 10` per process. These services typically run as Vercel functions or Kubernetes deployments — both are long-enough-lived that pool reuse pays off.
- web (Next.js): mixed. Server actions and route handlers use `postgres-js` (longer-lived); edge-runtime routes use `neon-http`. The `db/index.ts` exports a single `db` that delegates based on the runtime detection.

Per-instance pool size: **start at `max: 10`**. Neon free tier allows 100 concurrent connections; with 5 services × 2 instances × 10 = 100, that's the upper bound. Pro tier raises this. When N>2 instances per service is on the table, the pool size should drop to 5 — Postgres connection count is the constraining resource, not throughput.

Idle timeout: 30 seconds (Vercel function lifetime hint; warm-start tries to reuse first).

Statement timeout: 5 seconds default; 30 seconds for the audit-checkpoint signer (it reads + signs); 60 seconds for nightly cron jobs. Set in app code per-transaction, not as a database global.

## 9. Transaction semantics in Drizzle

```ts
// Single-statement query — no transaction needed.
await db.insert(sessions).values(record)

// Multi-statement transaction — default isolation (READ COMMITTED on pg).
await db.transaction(async (tx) => {
  await tx.update(sessions).set({ idleExpiresAt }).where(eq(sessions.id, id))
  await tx.insert(auditLog).values(entry)
})

// Isolation override — SERIALIZABLE for hardening-critical paths.
await db.transaction(
  async (tx) => {
    const row = await tx.select({ totalActions, recentTimestamps })
      .from(sessionActionCount)
      .where(eq(sessionActionCount.sessionId, sessionId))
      .for('update')
      .limit(1)
    // ... decide and update ...
  },
  { isolationLevel: 'serializable' },  // Drizzle passes this through
)
```

Drizzle's `transaction(..., { isolationLevel })` is dialect-aware. For SQLite it's a no-op (SQLite has only the implicit serialized writer). For pg-core it emits `BEGIN ISOLATION LEVEL SERIALIZABLE`. The same call site works on both backends — call-site code doesn't change between dialects.

The `.for('update')` row-lock hint exists in `drizzle-orm/pg-core` but not `drizzle-orm/sqlite-core`. The portable approach: define a `withRowLock(query)` helper in `packages/sdk/src/db/locks.ts` that no-ops on SQLite and emits `FOR UPDATE` on pg. PR-1 introduces this seam.

Retry helper:

```ts
export async function withSerializableRetry<T>(
  fn: () => Promise<T>,
  { maxAttempts = 3, baseDelayMs = 5 }: { maxAttempts?: number; baseDelayMs?: number } = {},
): Promise<T> {
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try { return await fn() } catch (err) {
      if (isSerializationFailure(err) && attempt < maxAttempts) {
        await sleep(baseDelayMs * 2 ** (attempt - 1))
        continue
      }
      throw err
    }
  }
  throw new Error('unreachable')
}
```

`isSerializationFailure(err)` checks `err.code === '40001'` (postgres-js, neon) or `err.message.match(/could not serialize/)`. On SQLite this never fires — the helper is a no-op pass-through.

## 11. Operator setup

### Per-service database

One Postgres database per service is the recommendation:

- `smart_agent_a2a` — a2a-agent
- `smart_agent_person_mcp` — person-mcp
- `smart_agent_org_mcp` — org-mcp
- `smart_agent_hub_mcp` — hub-mcp
- `smart_agent_web` — web

Blast-radius isolation: a runaway query in one service doesn't queue up in another. Per-service permissions: each service's IAM-equivalent (Neon role) gets `SELECT/INSERT/UPDATE/DELETE` only on its own database. Per-service backup schedules and PITR settings.

The shared-database alternative (one DB, per-service schemas) is acceptable for cost reasons in dev/staging but should not be the prod default. The marginal Neon-pro cost of N databases is ~$25/mo per — the operational simplification is worth it.

### Connection strings

Each service reads `DATABASE_URL` from env. Neon-Marketplace integration sets these per-environment (Production / Preview / Development) automatically; the CI runner gets `DATABASE_URL` pointing at the docker-compose Postgres.

```bash
# Production (Neon, OIDC-injected by Vercel Marketplace)
DATABASE_URL=postgres://...neon.tech/smart_agent_person_mcp?sslmode=require&connect_timeout=5

# Preview (Neon branch per PR)
DATABASE_URL=postgres://...neon.tech/smart_agent_person_mcp_pr_123?sslmode=require

# Local dev (SQLite, default; pg if a developer wants to mirror prod)
DATABASE_URL=sqlite:./person-mcp.db
# or
DATABASE_URL=postgres://postgres:postgres@localhost:5432/smart_agent_person_mcp
```

### Migration tool

`pnpm drizzle-kit generate` at change-time (developer commits the SQL file). `pnpm drizzle-kit migrate` at deploy time (CI runs it before flipping traffic to the new instance). The lockfile-style `meta/_journal.json` per service guarantees applied-only-once semantics across re-deploys.

Migrations directory: `apps/<svc>/drizzle/` (already in use for web).

### Backups

Neon Pro retains 7 days of PITR automatically; that's the prod RPO target. For longer retention, a nightly `pg_dump` to S3 from a Vercel cron is acceptable for v1; promote to a managed backup product (Neon's longer-retention tier, or a dedicated tool) when the data volume justifies it.

### Monitoring

Per service, alert on:
- Connection-pool exhaustion (`pool.totalCount === pool.options.max` sustained > 30s)
- `40001` retry rate > 0.5% of relevant txns
- p99 query latency > 200ms
- Replication lag (if/when read replicas land — §15)

## 13. Risk

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Schema drift between SQLite + Postgres dialects | Medium | Medium — bugs only manifest on the non-tested backend | `scripts/check-schema-parity.ts` lints. CI runs both backends until PR-6. |
| Migration order error (sessions before audit_log) | Low | High — boot fails | drizzle-kit's deterministic migration order; CI smoke test boots a fresh DB end-to-end |
| Test-environment complexity (containers slow CI) | Medium | Low | `pg-mem` for unit tests (in-memory pg-protocol mock); `testcontainers` for integration; matrix only on PRs touching db/* |
| Local-dev story: dev wants zero-config | Certain | Medium — every developer-day affected | Keep SQLite as default for `DATABASE_URL=sqlite:./local.db` (the unset case). `pnpm dev` stays zero-Postgres. Document `pnpm dev:pg` as the opt-in. |
| Cold-start latency on Vercel functions | Medium | Low | Neon HTTP driver for edge routes; `postgres-js` pool for server routes; warm-start reuse |
| `40001` retry storms under load | Low | Medium — hot-account contention | `withSerializableRetry` caps at 3; exponential backoff; surface as `action-cap-exceeded` if it persists (semantically correct: contention means cap is being hit) |
| Connection-count ceiling at scale | Low (today) | High at scale | Per-service `max: 10`; document the formula `services × instances × max ≤ DB ceiling`; surface as a dashboard panel |
| Drizzle dialect feature parity (jsonb operators) | Low | Low | Confine JSON manipulation to app code v1; revisit moving to `jsonb_*` after PR-6 |
| Bypass-check regression | Low | Medium | `scripts/check-no-bypass.sh` extended per-PR to track new patterns; CI fail-on-missing |

The local-dev SQLite preservation is the meaningful one for developer velocity. Dropping it would force every contributor to run Postgres locally, which is friction for one-off tasks. Keeping it as the default-when-`DATABASE_URL`-unset is the canonical "works on my laptop" path; opting in to `pnpm dev:pg` is for contributors actively working on storage code. The trade-off: a tiny dialect bug can hide for a feature cycle until pg-only tests catch it. The schema-parity lint plus the always-pg CI matrix on storage-touching PRs is the trip-wire.

## 14. Acceptance criteria

The merge gate for PR-5 (the last service to flip):

- `pnpm typecheck` clean against both backends.
- `pnpm test --backend=sqlite` and `pnpm test --backend=postgres` both pass.
- New integration test in `apps/person-mcp/test/multi-instance.pg.test.ts`: spin two `postgres-js` pools against the same DB, simulate two instances. Specifically:

  - **Audit-chain consistency**: 10 parallel `appendAuditEntry` for the same account across two pools, the resulting `audit_log` rows form a single linear chain (each row's `prev_entry_hash` matches the previous row's `entry_hash`, ordered by `seq`). No forks.
- The S3.1 checkpoint signer, running on instance A, signs a head that includes rows written by instance B.
- Operator runbook updated: `docs/runbooks/postgres-operations.md` covers connection-pool exhaustion, `40001` retry-rate alerts, PITR restore drill.

## 15. Out of scope

Explicitly **not** in S3.3:

- **Multi-region replication**. Single-region Postgres is enough until the product crosses geography. Neon's read-replica + write-region story is the planned follow-up.
- **Read replicas**. Read traffic in v1 is small compared to write traffic on the hot paths (sessions, audit). When read p95 climbs, a per-service replica is a config flag in Neon and a small change to a routing helper.
- **Column-level encryption at rest** for sensitive cells (`grant_json`, `holder_wallets.link_secret_id`, `encrypted_package`). Neon provides transparent storage encryption (AES-256 at the volume layer); `pgcrypto` for per-cell encryption is a Phase-2 task tracked alongside the KMS migration (K3/K4 — see `output/KMS-IMPLEMENTATION-PLAN.md`).
- **ORM swap**. Drizzle stays. Prisma, Kysely, raw SQL — all rejected because Drizzle's dialect-portability is exactly the feature this migration depends on.
- **Event-sourcing the audit log**. The S3.1 hash chain + Postgres BIGSERIAL is sufficient; full Kafka/etc. event-sourcing is over-engineering for v1.
- **Sharding**. Single-database-per-service is the v1 model; sharding (e.g. `audit_log` by `smart_account_address`) is a v3+ concern.
- **Migrating Askar SSI storage**. The Askar wallet store (`apps/person-mcp/src/ssi/storage/askar.ts`) is a separate substrate (its own embedded SQLite under the hood, owned by the Aries library). Treat as black-box; out of scope.
